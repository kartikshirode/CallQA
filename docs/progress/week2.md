# Week 2 results: ASR and WER

Goal for the week was the first real metric on top of the Week 1 dataset:
transcribe both tiers with local Whisper, score WER and CER against the gold
transcripts, and report latency and cost. The code is built, tested, and
committed. The GPU sweep that fills in the actual numbers is held on purpose and
runs when we say go, so the results table below is marked pending until then.

## What got built

The ASR stack follows the same pure-versus-heavy split as the diarization code,
so almost all the logic is tested with no GPU.

- `callqa/asr/normalize.py`, the WER normalizer. Lowercases, drops the
  HarperValley bracketed markers like [noise] and <unk>, strips punctuation, and
  joins contractions so that's and thats read the same. This is the module that
  decides whether real WER is honest, so it carries the widest test coverage.
- `callqa/asr/metrics.py`, WER and CER on jiwer, per call and pooled across a
  corpus into one AsrScore. Has an empty-reference guard so a silent reference
  never divides by zero.
- `callqa/asr/transcript.py`, the provider-neutral Transcript object with a JSON
  round-trip for the on-disk cache.
- `callqa/asr/whisper_adapter.py`, the faster-whisper wrapper. Loads a model size
  on cuda once and caches it, same shape as the pyannote adapter.
- `callqa/asr/providers.py`, the AsrProvider interface plus the local Whisper
  provider and the Deepgram and AssemblyAI stubs. The stubs raise a Week 6 note,
  so no dead cloud code ships early.
- `scripts/asr_benchmark.py`, the sweep. Transcribes the registry across
  tiny/base/small.en, caches each transcript to data/asr, scores per tier, and
  writes docs/asr_benchmark_report.md.

## Numbers that matter

- Tests: 109 passing, all offline. Six stage files now, Week 2 added test_asr.
- Model sweep: tiny.en, base.en, small.en. Full sweep over the 60 calls lands
  near 10-12 minutes on the 4060, cached after the first run.
- WER, CER, latency per audio minute, and a processing-time cost model, per call
  and pooled per tier.

The benchmark report and its WER and CER table are generated by the sweep, so
they are pending the run. As a baseline expectation, the Week 1 spot check with
base.en put synthetic WER at 0 to 4% and HarperValley at 21 to 35%. That gap is
the point: synthetic is clean, so the credible real-world WER comes from
HarperValley. The full sweep replaces those spot numbers with per-tier pooled
figures across three model sizes.

## Decisions made this week

- Full model sweep over a single baseline. Three sizes cost about ten minutes
  once and give the accuracy versus latency tradeoff, which is the story the
  project is pitching, not a single WER number.
- Contractions join rather than split in the normalizer. Human transcripts and
  Whisper often disagree on apostrophe style, and splitting that's into two
  tokens would score the disagreement as a real error. Whisper's own normalizer
  joins them, so we match that.
- Cloud providers stubbed now, implemented in Week 6. The interface is defined so
  Deepgram and AssemblyAI slot in later without reshaping the caller, but nothing
  makes a network call yet.
- Synthetic WER is measured on the telephone-band audio, not the clean studio
  version, so the number reflects the degraded 8 kHz signal.
- Transcripts cache to disk, so re-running the report never re-hits the GPU.

## What is left

The GPU sweep. Run `python scripts/asr_benchmark.py` on a machine with the 4060
to fill in docs/asr_benchmark_report.md. The live decode path in the adapter is
the one piece the offline suite cannot exercise, so the run is also where it gets
its first real check. All audio is on disk, so no rebuild is needed first.
